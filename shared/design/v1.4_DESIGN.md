## 1. 版本定位

### 1.1 v1.4 要解决什么问题

v1.3 完成了 Fragment Shader ISA 执行框架（36 条指令、ShaderCore、WarpScheduler、MemorySubsystem），但遗留了 **5 个严重的结构性缺陷**（见 `DESIGN_IMPROVEMENT_v1.2_FIX.md`），导致：

1. **FragmentShader 未真正使用 WarpScheduler** — WarpScheduler 框架在 v1.3 已完整实现，但 FragmentShader 的主循环是 `for (const auto& frag : fragments)` 串行调用 `executeFragment`，WarpScheduler 存在但形同虚设，fragment 仍串行执行，ShaderCore 无法发挥 SIMD/Warp 并行优势
2. **DIV 指令周期模拟缺失** — 实现已用牛顿迭代（2轮，精度达标），但声明 7 周期却无延迟模拟，实际 ~1 周期完成，性能数据不可信
3. **TokenBucket 带宽模型部分失效** — per-tile 接口（`storeTileToGMEM`/`loadTileFromGMEM`）已正确检查返回值；但 `storeAllTilesFromBuffer` 无条件 memcpy 忽略返回值；且 `TokenBucket::refill()` 直接填满 tokens 而非基于墙上时间线性补充
4. **L2 Cache 128KB 对 4.9MB 帧缓存命中率极低** — 约 1.3%，无法有效减少 GMEM 访问（设计目标 256KB 仍未达到）
5. **ISA 缺少 1 条高频指令** — DP3（opcode 0x25，SMOOTHSTEP 已占用 0x24），限制了着色器程序表达能力

此外，路线图承诺的 v1.4 功能中，**Early-Z ✅** 和 **纹理采样单元骨架 ✅** 已覆盖，但 **MSAA ❌** 和 **Write Buffer ❌** 尚未设计。

## 2. 架构改动

### 2.1 核心设计决策

**决策 1：FragmentShader 必须真正使用 WarpScheduler**

v1.3 中 WarpScheduler 已完整实现框架，但 FragmentShader 的主循环是 `for (const auto& frag : fragments)` 串行调用 `executeFragment`。v1.4 必须将 Warp 批处理接入，使多个 fragment 真正并行执行。

> **⚠️ WarpScheduler v1.3 状态澄清：**  
> README.md 路线图中 WarpScheduler 标注为 ✅ 100% 完成，这是**文档标记正确性**问题：v1.3 确实完成了 WarpScheduler 的**框架实现**（WarpScheduler 类、调度逻辑、TileBuffer 接口），但**未将其接入 FragmentShader 的执行路径**。因此：  
> - 不是说 README 撒谎，而是"框架完成"≠"功能集成完成"  
> - v1.3 的 WarpScheduler 是独立的、未被调用的模块  
> - v1.4 P0-1 的定性：从严格语义上讲是"遗留未完成项接入"而非"回归修复"（没有东西坏了，只是没做完），但由于优先级和影响等同于 P0 回归，保持 P0-1 分类不变，更新描述为"**P0-1 WarpScheduler 接入 FragmentShader（遗留未完成项）**"

**决策 2：Early-Z 放在 FragmentShader 之前**

传统 GPU 的 Early-Z 优化在 Rasterizer 输出 fragment 后、FS 执行前做深度测试，拒绝掉被遮挡的 fragment。v1.4 在 `Rasterizer → FragmentShader` 之间插入 Early-Z 阶段，减少无效着色。

**决策 3：TEX 指令返回 placeholder，不实现完整纹理采样管线**

完整纹理采样需要纹理缓存、UV 变换、MIPmap 选择等，v1.4 仅骨架实现：TEX 指令执行后返回 `(u, v)` 对应的 placeholder 颜色（checkboard pattern），保证 ISA 完整性验证可运行。

**决策 4：L2 Cache 增大到 256KB，替换策略改为 tile-aware**

128KB → 256KB，set × way 从 256×8 改为 512×8。命中热点 tile 时优先保留，不被跨 tile 访问挤出。

**决策 5：MSAA 在 Rasterizer 阶段实现多重采样**

MSAA 在光栅化阶段为每个 pixel 生成 N 个 sample 点（v1.4 支持 2x 和 4x），在 Early-Z 通过后，每个 sample 独立执行 FS（如果该 sample 通过深度测试），最后在 Framebuffer 合并。

**决策 6：Write Buffer 减少 GMEM 访问次数**

在 Framebuffer 的 tile 数据写回 GMEM 前，暂存于 Write Buffer（一个 tile-sized 缓存），批量累积后一次性写回，摊薄固定开销。

### 2.2 管线阶段变化

```
v1.3 管线:
CommandProcessor → VertexShader → PrimitiveAssembly → TilingStage
→ Rasterizer → FragmentShader → Framebuffer → TileWriteBack

v1.4 管线（变化点用 [NEW]/[MOD] 标注）:
CommandProcessor → VertexShader → PrimitiveAssembly → TilingStage
→ Rasterizer [MOD: MSAA采样] → Early-Z [NEW] → FragmentShader [MOD: Warp集成]
→ Framebuffer [MOD: WriteBuffer] → TileWriteBack
```

## 4. 模块设计

### 4.1 FragmentShader Warp 集成（P0-1）

**文件：** `src/stages/FragmentShader.cpp`

**核心改动：** `execute()` 主循环从逐 fragment 改为以 warp=8 为批次：

```cpp
// FragmentShader.cpp — execute() 修改
void FragmentShader::execute() {
    const uint32_t WARP_SIZE = 8;
    const auto& fragments = m_inputFragments;

    for (size_t base = 0; base < fragments.size(); base += WARP_SIZE) {
        if (base >= fragments.size()) break;  // 无符号回绕 guard

        size_t batch_size = std::min(static_cast<size_t>(WARP_SIZE),
                                      fragments.size() - base);

        // 1. 构建 warp fragment 上下文（最多 8 个）
        std::vector<FragmentContext> warp_frags;
        warp_frags.reserve(batch_size);
        for (size_t i = 0; i < batch_size; ++i) {
            FragmentContext ctx = fragmentToContext(fragments[base + i]);
            ctx.tile_x = m_tileX;
            ctx.tile_y = m_tileY;
            warp_frags.push_back(ctx);
        }

        // 2. 调用 ShaderCore 的 Warp 批处理（真正使用 WarpScheduler）
        m_shaderCore.executeWarpBatch(warp_frags, m_shaderFunction,
                                       m_warpScheduler);

        // 3. 收集结果，写入 TileBuffer
        for (size_t i = 0; i < warp_frags.size(); ++i) {
            const auto& ctx = warp_frags[i];
            if (ctx.killed) continue;  // depth test killed

            Fragment shaded = contextToFragment(ctx);
            uint32_t lx = fragments[base + i].x - m_tileX * TILE_WIDTH;
            uint32_t ly = fragments[base + i].y - m_tileY * TILE_HEIGHT;

            float color[4] = {shaded.r, shaded.g, shaded.b, shaded.a};
            m_tileBuffer->depthTestAndWrite(m_tileIndex, lx, ly, shaded.z, color);
        }
    }
}
```

**注意：** `executeWarpBatch()` 需在 `ShaderCore` 中实现，复用 `Interpreter` 执行每条 ISA 指令（而非 `executeFragmentInternal()` 的简化路径）。

### 4.2 DIV 指令周期延迟模拟（P0-2）

**文件：** `src/core/ExecutionUnits.hpp` + `src/isa/Interpreter.cpp`

**背景：** 牛顿迭代算法（2轮 Newton-Raphson，精度 < 1e-4）已正确实现。真正问题：SFU/Interpreter 层声明 7 周期但未实际累加 `cycle_count`，导致 DIV 在性能模型中约 ~1 周期完成，与硬件行为不符。

**修复方式：** 在 DIV 执行路径上，累加 `stats_.cycles += 7`（或实际迭代轮数对应的周期数），使性能数据真实可信。

### 4.3 TokenBucket 带宽模型修复（P0-3）

**文件：** `src/core/MemorySubsystem.cpp` + `src/stages/TileWriteBack.cpp`

**真正问题（两项独立）：**

1. **`storeAllTilesFromBuffer` 无条件 memcpy**：当前实现直接 memcpy 所有 tile 数据到 GMEM，完全忽略 `addAccess` 返回值，导致带宽模型在批量写回路径上形同虚设。需在批量写回前检查带宽可用量，不足时模拟 stall（记录等待时间到 `cycle_count`）。

2. **`TokenBucket::refill()` 直接填满 tokens**：当前 refill 在每次调用时直接将 tokens 填满到 `maxTokens`，而非基于墙上时间线性补充。这意味着带宽补充速度与实际时间脱钩，headless benchmark 中时间不流逝时 tokens 永远满额。修复为：

```cpp
// MemorySubsystem.cpp — TokenBucket::refill() 修复
void TokenBucket::refill() {
    double now = getCurrentTimeNs() / 1e9;  // 秒
    double elapsed = now - lastRefillTime;   // 秒
    double newTokens = elapsed * refillRate;  // bytes
    tokens = std::min(maxTokens, tokens + newTokens);
    lastRefillTime = now;
}
```

**注意：** per-tile 接口（`storeTileToGMEM`/`loadTileFromGMEM`）已正确检查 `addAccess` 返回值，无需修改。

### 4.4 L2 Cache 256KB + tile-aware 替换（P0-4）

**文件：** `src/core/MemorySubsystem.hpp`

**当前状态：** 128KB（256 sets × 8 ways × 64B）  
**目标：** 256KB（512 sets × 8 ways × 64B）

```cpp
// MemorySubsystem.hpp — L2 Cache 配置
constexpr uint32_t L2_CACHE_SETS   = 512;  // 从 256 增大到 512
constexpr uint32_t L2_CACHE_WAYS   = 8;   // 保持 8 way
constexpr uint32_t L2_CACHE_LINE_SIZE = 64;  // bytes
// 256 KB = 512 × 8 × 64 = 262,144 bytes
```

**tile-aware 替换策略：** 同一 tile 热点数据互相保护，不同 tile 之间才做 LRU 替换。

### 4.5 Early-Z（P1-2）

**新增文件：** `src/stages/EarlyZ.hpp` + `src/stages/EarlyZ.cpp`

```cpp
// EarlyZ — FragmentShader 前的深度预测试
class EarlyZ {
public:
    void setDepthBuffer(const float* depthBuffer) { m_depthBuffer = depthBuffer; }
    void setDepthTestEnabled(bool enabled) { m_enabled = enabled; }

    // 对单 pixel 做 early depth test
    // 返回 true = 通过测试（需要继续 FS）；false = 被遮挡（跳过 FS）
    bool test(uint32_t x, uint32_t y, float z) const {
        if (!m_enabled) return true;
        size_t idx = y * WIDTH + x;
        return z < m_depthBuffer[idx];
    }

    // 对一批 fragments 做 early depth test，返回通过的 fragments
    std::vector<Fragment> filter(const std::vector<Fragment>& fragments) const;

private:
    const float* m_depthBuffer = nullptr;
    bool m_enabled = true;
};
```

**集成方式：** `RenderPipeline::executeTile()` 中，Rasterizer 输出 fragments 后立即调用 EarlyZ filter，再送入 FragmentShader。

### 4.7 MSAA — 多重采样抗锯齿（P2-4）

#### 4.7.1 设计目标

v1.4 支持 **2x MSAA** 和 **4x MSAA** 两种模式，在 Rasterizer 阶段为每个 pixel 生成 N 个 sample 点，每个 sample 独立经过 Early-Z 和 FS，最后在 Framebuffer 合并。

#### 4.7.2 MSAA 采样模式

**2x MSAA（2 samples per pixel）：**
```
Sample 位置（相对 pixel 中心，偏移量）:
  S0: (-0.25, -0.25)
  S1: (+0.25, +0.25)
```

**4x MSAA（4 samples per pixel）— 标准 RGSS：**
```
Sample 位置（相对 pixel 中心，偏移量）:
  S0: (-0.375, -0.125)
  S1: (+0.125, -0.375)
  S2: (-0.125, +0.375)
  S3: (+0.375, +0.125)
```

#### 4.7.3 管线集成

```
Rasterizer [MOD: MSAA]
  ↓ 每个 pixel 输出 N 个 sample fragment（而非 1 个）
  ↓
EarlyZ [MOD: per-sample depth test]
  ↓ 每个 sample 独立深度测试
  ↓
FragmentShader [MOD: per-sample shading]
  ↓ 每个通过 Early-Z 的 sample 独立执行 FS
  ↓
Framebuffer [MOD: MSAA resolve]
  ↓ 对 N 个 sample 颜色做合并（Resolve）
  ↓
WriteBuffer → GMEM
```

#### 4.7.4 数据结构变更

```cpp
// Fragment — 新增 sample 偏移量
struct Fragment {
    uint32_t x, y;      // pixel 坐标
    float z;            // 深度
    float w;            // 齐次坐标
    float u, v;         // 插值 UV
    uint32_t tri_id;    // 所属三角形

    // MSAA: 所属 sample 点索引（0..msaa_level-1）
    uint32_t sample_id;
    // MSAA: 该 sample 相对于 pixel 中心的偏移
    float sample_offset_x;
    float sample_offset_y;
};

// MSAA 配置
enum class MSAALevel { None = 1, x2 = 2, x4 = 4 };
struct MSAAResolve {
    // Resolve 策略：均值覆盖
    static float resolveDepth(const std::vector<float>& sampleDepths) {
        // 返回最小深度（保守策略，最小深度保证正确性）
        return *std::min_element(sampleDepths.begin(), sampleDepths.end());
    }

    static void resolveColor(float outColor[4],
                              const std::vector<std::array<float,4>>& sampleColors) {
        // 均值合并
        float r = 0, g = 0, b = 0, a = 0;
        for (const auto& c : sampleColors) {
            r += c[0]; g += c[1]; b += c[2]; a += c[3];
        }
        float n = static_cast<float>(sampleColors.size());
        outColor[0] = r / n; outColor[1] = g / n;
        outColor[2] = b / n; outColor[3] = a / n;
    }
};
```

#### 4.7.5 MSAA Depth Buffer 布局

MSAA per-sample 深度数据存储在 `m_sampleDepthBuffer` 中：

- **数组大小：** `width × height × msaa_level` 个 float
- **索引公式：** `idx = (y * width + x) * msaa_level + sample_id`
  - `(x, y)`：像素坐标
  - `sample_id`：该像素内的 sample 索引（0 .. msaa_level-1）

```
示例（4x MSAA，像素 (3, 5) 的 S2 sample）：
  idx = (5 * WIDTH + 3) * 4 + 2
```

每个 sample 的深度在 Rasterizer 阶段插值得到，独立通过 Early-Z 测试后送入 FS 着色。Framebuffer resolve 时使用 `MSAAResolve::resolveDepth()` 取最小深度。

### 4.8 Write Buffer — 批量写缓冲（P2-5）

#### 4.8.1 设计目标

减少 GMEM 写操作次数。当前每处理完一个 tile 就立即写回 GMEM，频繁的小数据写回会产生大量固定开销（地址译码、总线仲裁等）。Write Buffer 暂存多个 tile 的数据，批量累积后一次性写回。

#### 4.8.2 架构

```
Framebuffer                    Write Buffer                 GMEM
  │                              │                           │
  │  writeTile(tile_data)        │                           │
  │─────────────────────────────►│                           │
  │                              │  [累积 tile A]            │
  │                              │                           │
  │  writeTile(tile_data)        │                           │
  │─────────────────────────────►│  [累积 tile B]            │
  │                              │                           │
  │  writeTile(tile_data)        │                           │
  │─────────────────────────────►│  [累积 tile C]            │
  │                              │                           │
  │     [flush threshold reached]│                           │
  │                              │  flush()                  │
  │                              │─────────────────────────►│ [一次性写回 A,B,C]
  │                              │  [buffer cleared]         │
```

#### 4.8.3 WriteBuffer 类设计

**新增文件：** `src/stages/WriteBuffer.hpp` + `src/stages/WriteBuffer.cpp`

```cpp
// WriteBuffer.hpp
class WriteBuffer {
public:
    // tile_size: 每个 tile 的字节数（如 256×256×4 = 262144 bytes）
    // capacity:  buffer 中最多缓存的 tile 数量（默认 4）
    // flush_threshold: 达到该数量时自动 flush（默认 4）
    WriteBuffer(size_t tile_size, size_t capacity = 4, size_t flush_threshold = 0);

    // 写入一个 tile 的数据（异步，暂存 buffer）
    // 返回 true = 成功暂存；false = buffer 已满（需要先 flush）
    bool writeTile(const void* tile_data);

    // 强制将 buffer 中所有暂存的 tile 一次性写回 GMEM
    // 消耗带宽，会 stall
    void flush();

    // 查询
    size_t numPendingTiles() const { return m_pendingTiles.size(); }
    bool isEmpty() const { return m_pendingTiles.empty(); }

    // 带宽感知：当 TokenBucket 带宽不足时 stall
    void setBandwidthLimiter(TokenBucket* limiter) { m_bandwidthLimiter = limiter; }

private:
    std::vector<std::vector<uint8_t>> m_pendingTiles;  // 暂存的 tile 数据
    size_t m_tileSize;
    size_t m_capacity;
    size_t m_flushThreshold;   // 0 表示不自动 flush，只手动 flush
    TokenBucket* m_bandwidthLimiter = nullptr;
};
```

#### 4.8.4 刷新策略

```cpp
// WriteBuffer.cpp
void WriteBuffer::flush() {
    if (m_pendingTiles.empty()) return;

    // 带宽检查：如果设置了带宽限制，分摊到每个 tile 检查
    if (m_bandwidthLimiter) {
        size_t totalBytes = m_pendingTiles.size() * m_tileSize;
        // 模拟带宽不足 stall
        while (totalBytes > 0) {
            size_t access = std::min(totalBytes, m_tileSize);
            bool ok = m_bandwidthLimiter->addAccess(access);
            if (!ok) {
                // 带宽不足，记录 stall 周期
                stats_.cycles += 10;  // stall 惩罚周期
                continue;
            }
            totalBytes -= access;
        }
    }

    // 批量 memcpy 到 GMEM（一次大操作替代多次小操作）
    for (const auto& tile : m_pendingTiles) {
        GMEM::getInstance().write(tile.data(), tile.size());
    }

    m_pendingTiles.clear();
}
```

#### 4.8.5 集成到 Framebuffer/TileWriteBack

**文件：** `src/stages/Framebuffer.cpp` 或 `src/stages/TileWriteBack.cpp`

```cpp
// TileWriteBack.cpp — 使用 WriteBuffer
void TileWriteBack::storeTileToGMEM(uint32_t tile_index, const TileData& tile) {
    // 不再直接写 GMEM，而是写入 WriteBuffer
    bool ok = m_writeBuffer->writeTile(tile.data());
    if (!ok) {
        // buffer 满了，强制 flush 后再写
        m_writeBuffer->flush();
        m_writeBuffer->writeTile(tile.data());
    }

    // 可选：定期检查是否达到 flush threshold
    if (m_writeBuffer->numPendingTiles() >= m_flushThreshold) {
        m_writeBuffer->flush();
    }
}

// 每帧结束时（frame boundary）强制 flush 所有残留 tile
void TileWriteBack::endFrame() {
    m_writeBuffer->flush();
}
```

#### 4.8.6 性能收益估算

假设一帧有 300 个 tile（10×10 grid，每 tile 256×256），每个 tile 262KB：

| 策略 | GMEM 写操作次数 | 总字节数 |
|------|----------------|----------|
| 无 WriteBuffer（每 tile 单独写） | 300 次 | 78.6 MB |
| WriteBuffer（4 tile 批量） | 75 次 | 78.6 MB |
| WriteBuffer（8 tile 批量） | 38 次 | 78.6 MB |

固定开销降低约 **4–8 倍**，具体收益取决于 GMEM 控制器的地址译码和总线仲裁开销。

### 4.9 WarpScheduler Debug Toggle 设计（P0-1 辅助）

详细内容见 [`v1.4_WARP_DEBUG_DESIGN.md`](v1.4_WARP_DEBUG_DESIGN.md)。

#### 设计原则

单线程环境下 `parallel` 和 `serial` 完全没区别，复杂的多模式设计（parallel/serial/auto/verbose）价值有限。真正有价值的只有一个：**batch size 可配**，方便 debug 时切小批次验证。

#### 命令行接口

| Flag | 行为 | 默认值 |
|------|------|--------|
| `--warp` | 开启 warp batch | 关闭（串行） |
| `--warp-size=N` | 配置 batch size（可配 4/8/16） | 8 |

#### 示例

```bash
# 默认串行（等价 v1.3）
./softgpu --input=scene.scn

# 开启 warp batch，默认 size=8
./softgpu --warp --input=scene.scn

# 开启 warp batch，size=4（方便调试单步跟踪）
./softgpu --warp --warp-size=4 --input=scene.scn
```

单线程环境下，无论 batch size 是 4 还是 8，执行路径完全相同（顺序执行 ISA 解释器），差异仅在于分批粒度。使用 `--warp-size=4` 减少调试时的 fragment 数量，便于单步跟踪。

#### 不提供的接口（已简化）

- ~~`--warp-mode=serial|parallel|auto`~~ — 多模式区分已删除
- ~~`--no-warp`~~ — 不需要，默认即串行
- ~~auto 降级~~ — 不需要，跑出什么结果就是什么结果
- ~~verbose 模式信息~~ — 不需要

### 5.4 性能基准测试

**当前机器限制：** 2 线程环境无法验证真实并行 Warp 性能。WarpScheduler 的并行收益（batch 调度减少调度开销、Early-Z 减少无效着色）需要在多核机器（8+ 核）上建立基准数据。

**v1.4 性能收益来源：** v1.4 的性能改进主要来自**架构改进**而非并行化：
- Early-Z：在 Rasterizer → FS 之间前置深度测试，跳过被遮挡 fragment 的着色开销
- Batch 调度：WarpScheduler 将 8 个 fragment 打包调度，减少解释器调度粒度开销
- Write Buffer：批量写回 GMEM，减少地址译码和总线仲裁固定开销
- DIV 周期精确模拟：消除虚假性能数据，使性能分析更可信

**基准数据建立计划：** 在 8+ 核多核机器上建立性能基准 CSV，包含：
- `warp_execution_time`：Warp 批处理总时间
- `shader_bound_score`：FS 瓶颈比例
- `early_z_kill_rate`：Early-Z 过滤掉的 fragment 比例
- `write_buffer_flush_count`：Write Buffer 刷新次数
