## R-P0-1：Benchmark 硬编码时间估算

### 问题

**文件：** `src/benchmark/BenchmarkRunner.cpp`（约第 136-139 行）

```cpp
// 简化：使用总时间减去估算的其他阶段
result.fragmentShaderTimeMs = elapsedMs * 0.6;  // 估算
result.tilingTimeMs = elapsedMs * 0.1;
result.rasterizerTimeMs = elapsedMs * 0.15;
result.tileWriteBackTimeMs = elapsedMs * 0.1;
```

各阶段时间以固定比例从总帧时间中估算，完全脱离真实 profiling 数据。

### 重构方案

利用 `FrameProfiler` 已收集的真实阶段耗时数据（通过 `beginStage/endStage` RAII 自动记录），替代硬编码系数。

**修改点 — `BenchmarkRunner::runSingleFrame()`：**

1. Pipeline 启用 profiler：`pipeline.setProfilerEnabled(true)`（需确认该接口存在，若不存在则仅在 `RenderPipeline` 内部 conditional）
2. 执行两次 `pipeline.render(command)`（预热 + 正式）
3. 从 `FrameProfiler::get().getStats(StageHandle::FragmentShader).ms` 等获取真实耗时
4. 回填 `result.fragmentShaderTimeMs` 等字段

**伪代码：**
```cpp
pipeline.render(command); // 预热（profiler warmup）
pipeline.getMemorySubsystem().resetCounters();
pipeline.render(command); // 正式（profiler 采集）

FrameProfiler& prof = FrameProfiler::get();
result.fragmentShaderTimeMs = prof.getStats(StageHandle::FragmentShader).ms;
result.tilingTimeMs        = prof.getStats(StageHandle::Tiling).ms;
result.rasterizerTimeMs    = prof.getStats(StageHandle::Rasterizer).ms;
result.tileWriteBackTimeMs = prof.getStats(StageHandle::TileWriteBack).ms;
```

> **注意：** `RenderPipeline` 目前通过手动 `beginStage/endStage` 记录，并非 ProfilerGuard RAII。`ProfilerGuard` 类在 codebase 中未找到，如后续需要可在 `FrameProfiler.hpp` 中补充。当前方案直接复用现有手动 API 即可满足需求。

### 风险评估

| 风险 | 等级 | 说明 |
|------|------|------|
| Profiler 未开启导致 0 值 | 低 | benchmark 场景默认启用 profiler，不启用时回退估算值 |
| 多帧累加影响基准测试精度 | 低 | 仅记录正式帧（一次），不开启额外循环 |

### 测试验证

- ✅ 运行 `BenchmarkRunner_test.cpp`，对比重构前后 `fragmentShaderTimeMs` 不为 0
- ✅ 对比同一场景重构前后 `speedup` 值波动 < 5%（功能未改变）

## R-P0-2：TileBuffer/TileWriteBack 逐像素循环

### 问题

**文件：**
- `src/stages/TileBuffer.cpp`：`loadFromGMEM()` / `storeToGMEM()` 内含逐像素 `for` 循环
- `src/stages/TileWriteBack.cpp`：`loadTileFromGMEM()` / `storeTileToGMEM()` / `loadAllTilesToBuffer()` / `storeAllTilesFromBuffer()` 内含大量逐像素循环

示例（TileBuffer.cpp）：
```cpp
for (uint32_t i = 0; i < TILE_SIZE; ++i) {
    tile.color[i * 4 + 0] = gmemColor[i * 4 + 0];
    tile.color[i * 4 + 1] = gmemColor[i * 4 + 1];
    tile.color[i * 4 + 2] = gmemColor[i * 4 + 2];
    tile.color[i * 4 + 3] = gmemColor[i * 4 + 3];
}
```

### 重构方案

将逐像素循环替换为 `std::memcpy`（同类型float数组，内存连续）：

**TileBuffer.cpp — `loadFromGMEM()`：**
```cpp
// Color: TILE_SIZE * 4 floats
std::memcpy(tile.color, gmemColor, TILE_SIZE * 4 * sizeof(float));
// Depth: TILE_SIZE floats
std::memcpy(tile.depth, gmemDepth, TILE_SIZE * sizeof(float));
```

**TileBuffer.cpp — `storeToGMEM()`：**
```cpp
std::memcpy(outColor, tile.color, TILE_SIZE * 4 * sizeof(float));
std::memcpy(outDepth, tile.depth, TILE_SIZE * sizeof(float));
```

**TileWriteBack.cpp — `loadTileFromGMEM()`：**
```cpp
std::memcpy(outTileBuffer.color,
            m_gmemColor.data() + colorOffset,
            TILE_SIZE * 4 * sizeof(float));
std::memcpy(outTileBuffer.depth,
            m_gmemDepth.data() + depthOffset,
            TILE_SIZE * sizeof(float));
```

**TileWriteBack.cpp — `storeTileToGMEM()`：**
```cpp
std::memcpy(m_gmemColor.data() + colorOffset,
            tileBuffer.color, TILE_SIZE * 4 * sizeof(float));
std::memcpy(m_gmemDepth.data() + depthOffset,
            tileBuffer.depth, TILE_SIZE * sizeof(float));
```

**TileWriteBack.cpp — `loadAllTilesToBuffer()`：**
```cpp
for (uint32_t tileIndex = 0; tileIndex < NUM_TILES; ++tileIndex) {
    auto& tile = manager.getTileBuffer(tileIndex);
    size_t colorOffset = getTileColorOffset(tileIndex);
    size_t depthOffset = getTileDepthOffset(tileIndex);
    std::memcpy(tile.color, m_gmemColor.data() + colorOffset, TILE_SIZE * 4 * sizeof(float));
    std::memcpy(tile.depth, m_gmemDepth.data() + depthOffset, TILE_SIZE * sizeof(float));
}
```

**TileWriteBack.cpp — `storeAllTilesFromBuffer()`：**
```cpp
for (uint32_t tileIndex = 0; tileIndex < NUM_TILES; ++tileIndex) {
    const auto& tile = manager.getTileBuffer(tileIndex);
    size_t colorOffset = getTileColorOffset(tileIndex);
    size_t depthOffset = getTileDepthOffset(tileIndex);
    std::memcpy(m_gmemColor.data() + colorOffset, tile.color, TILE_SIZE * 4 * sizeof(float));
    std::memcpy(m_gmemDepth.data() + depthOffset, tile.depth, TILE_SIZE * sizeof(float));
}
```

### 风险评估

| 风险 | 等级 | 说明 |
|------|------|------|
| 浮点 `memcpy` 语义问题 | 无 | float 数组是 POD 类型，memcpy 完全等价于逐元素拷贝 |
| `m_gmemColor.data()` 返回 `float*` vs `const float*` | 低 | `storeTileToGMEM` 写操作用 `data() + offset` 创建非常量指针，合法 |

### 测试验证

- ✅ `TileBuffer_test.cpp`（如存在）逐 tile 验证 depth/color 值不变
- ✅ `test_Integration.cpp` 端到端渲染结果像素级一致
- ✅ Benchmark `consumedBandwidthGBps` 在 memcpy 优化后可能有轻微变化（实际带宽降低），需确认性能数据合理性

## R-P0-3：RenderPipeline::render() 180行

### 问题

**文件：** `src/pipeline/RenderPipeline.cpp` 第 50-230 行，`render()` 函数违反单一职责原则，内含 TBR/PHASE1 两套逻辑分支、7个 stage 调用、profiling 代码混杂。

### 重构方案

将 `render()` 拆解为以下子函数：

| 新函数 | 职责 |
|--------|------|
| `renderTBRMode(command)` | PHASE2 TBR 完整流程 |
| `renderPhase1Mode(command)` | PHASE1 兼容模式流程 |
| `executePerTile(tileIndex, tileX, tileY)` | 单个 tile 的 load/rasterize/shade/store |
| `syncEmptyTilesToGMEM()` | 将空 tile bin 的 TileBuffer 内容同步到 GMEM |
| `syncGMEMToFramebuffer()` | GMEM → Framebuffer 像素拷贝（已有，可提取） |
| `updateFrameProfilingMetrics(command)` | 更新瓶颈检测指标（从 `updateBottleneckMetrics` 重命名并简化） |

**`render()` 变为调度层：**
```cpp
void RenderPipeline::render(const RenderCommand& command) {
    if (m_profilerEnabled) FrameProfiler::get().beginFrame();
    if (m_tbrEnabled) {
        renderTBRMode(command);
    } else {
        renderPhase1Mode(command);
    }
    if (m_profilerEnabled) {
        FrameProfiler::get().endFrame();
        updateFrameProfilingMetrics(command);
    }
}
```

**`executePerTile()` 签名：**
```cpp
void RenderPipeline::executePerTile(uint32_t tileIndex, uint32_t tileX, uint32_t tileY) {
    // 1. Load from GMEM (TileWriteBack)
    TileBuffer& tileMem = m_tileBuffer.getTileBuffer(tileIndex);
    m_tileWriteBack.loadTileFromGMEM(tileIndex, &m_memory, tileMem);

    // 2. Rasterizer (uses global tiling stage output; filter by tile)
    // 3. FragmentShader writes to TileBuffer
    // 4. Store back to GMEM
    m_tileWriteBack.storeTileToGMEM(tileIndex, &m_memory, tileMem);
}
```

> `executePerTile` 当前在 `render()` 内联实现，提取后逻辑不变。

### 风险评估

| 风险 | 等级 | 说明 |
|------|------|------|
| 状态泄露到拆出函数 | 低 | 使用成员变量已有良好封装，提取不改变可见性 |
| PHASE1 模式回归 | 低 | 提取后保留原有调用顺序和条件分支 |

### 测试验证

- ✅ 所有现有集成测试（`test_Integration.cpp`）通过
- ✅ `RenderPipeline` 单元测试（如存在）验证每阶段输出等价
- ✅ Benchmark 结果（FPS、带宽）对比重构前波动 < 2%

## R-P0-4：FragmentShader 裸指针 null check

### 问题

**文件：** `src/stages/FragmentShader.hpp` 第 66 行

```cpp
void setTileBufferManager(TileBufferManager* manager);
```

`setTileBufferManager()` 接收裸指针但无任何校验，若传入 `nullptr` 后调用 `m_tileBuffer->...` 将产生 UB。

### 重构方案

在 `.cpp` 实现文件中添加 null check + LOG_ERROR：

**FragmentShader.cpp（如 `setTileBufferManager` 在 cpp 中）或直接在 hpp 内联：**

```cpp
void FragmentShader::setTileBufferManager(TileBufferManager* manager) {
    if (!manager) {
        LOG_ERROR("FragmentShader: setTileBufferManager received nullptr");
        return;
    }
    m_tileBuffer = manager;
}
```

同时在 `execute()` 或 `setInputAndExecuteTile()` 中对 `m_tileBuffer` 的使用处添加防御性检查：

```cpp
void FragmentShader::setInputAndExecuteTile(const std::vector<Fragment>& fragments,
                                             uint32_t tileX, uint32_t tileY) {
    if (!m_tileBuffer) {
        LOG_ERROR("FragmentShader: TileBufferManager not set, cannot execute in tile mode");
        return;
    }
    // ... 正常逻辑
}
```

> `LOG_ERROR` 宏需确认存在于 codebase（如使用 `spdlog` 或自实现 `Logger`）。

### 风险评估

| 风险 | 等级 | 说明 |
|------|------|------|
| 现有调用链可能依赖 null 行为 | 低 | PHASE2 代码路径均保证 non-null 传入 |
| LOG_ERROR 引入 I/O 开销 | 可忽略 | 仅在 bug 触发时调用，非热路径 |

### 测试验证

- ✅ 构造 `FragmentShader` 不设置 `TileBufferManager` 并调用 `execute()`，确认不崩溃 + 打印 ERROR 日志
- ✅ 正常流程测试通过（null check 不触发）

## R-P0-5：MemorySubsystem 带宽模型失效

### 问题

**文件：** `src/core/MemorySubsystem.cpp` 第 135 行

```cpp
bool MemorySubsystem::addAccess(size_t bytes, MemoryAccessType type) {
    bool allowed = m_bucket.tryConsume(bytes);  // ← 返回值
    m_accessCount++;
    // ...
    return allowed;  // ← 返回 allowed，但调用方可能忽略
}
```

`tryConsume()` 返回 `bool` 表示令牌桶中是否有足够配额。问题在于：
1. 调用方忽略返回值（如某些路径不检查 `allowed`）
2. 带宽不足时访问仍被记录（`switch` 仍执行），导致带宽模型失效

### 重构方案

两种修复策略，**推荐策略 A**：

**策略 A（推荐）：严格模式 — 带宽不足时拒绝访问**

在 `addAccess()` 内部，当 `allowed == false` 时，不记录该次访问（返回后调用方不应再使用数据）：

```cpp
bool MemorySubsystem::addAccess(size_t bytes, MemoryAccessType type) {
    bool allowed = m_bucket.tryConsume(bytes);
    m_accessCount++;

    if (!allowed) {
        // 带宽不足：记录为 stall，但跳过数据统计
        m_bandwidthStallCount++;
        return false;
    }

    switch (type) {
        case MemoryAccessType::LoadTile:
            m_totalReadBytes += bytes;
            m_totalBandwidthBytes += bytes;
            break;
        case MemoryAccessType::StoreTile:
            m_totalWriteBytes += bytes;
            m_totalBandwidthBytes += bytes;
            break;
        case MemoryAccessType::ReadVertex:
            m_totalReadBytes += bytes;
            m_totalBandwidthBytes += bytes;
            break;
    }

    return true;
}
```

**策略 B（保守）：宽松模式 — 始终记录但标记 stall**

如果现有代码依赖"带宽不足时仍执行访问"的行为，则保持记录但添加 stall 计数和日志：

```cpp
if (!allowed) {
    m_bandwidthStallCount++;
    // 仍记录（保守），但后续可依 stall 比例降频
}
```

**关键：找到所有忽略 `addAccess()` 返回值的地方并修复。**

### 风险评估

| 风险 | 等级 | 说明 |
|------|------|------|
| 严格模式下已有代码依赖跳过检查导致功能破坏 | 中 | 需全面审查所有 `addAccess()` 调用点 |
| 策略 B 可能引入"虚假带宽"数据 | 低 | stall 计数帮助定位瓶颈，不影响正确性 |

### 测试验证

- ✅ 构造大量并发内存访问场景，验证 stall 计数 > 0 时系统仍正确运行
- ✅ `MemorySubsystem` 单元测试验证 `addAccess` 返回 false 时 `totalReadBytes` 不增长
- ✅ Benchmark 对比：带宽数据更准确反映真实消耗

## 附录：现有 Profiler API 摘要

```cpp
// 启用/禁用
pipeline.setProfilerEnabled(true);

// 阶段计时
FrameProfiler::get().beginFrame();
FrameProfiler::get().beginStage(StageHandle::FragmentShader);
FrameProfiler::get().endStage(StageHandle::FragmentShader);
FrameProfiler::get().endFrame();

// 读取结果
auto stats = FrameProfiler::get().getStats(StageHandle::FragmentShader);
double ms = stats.ms;           // 该阶段累计时间（ms）
uint64_t inv = stats.invocations; // 调用次数
```

`StageHandle` 枚举值包括：`CommandProcessor`, `VertexShader`, `PrimitiveAssembly`, `Tiling`, `Rasterizer`, `FragmentShader`, `Framebuffer`, `TileWriteBack`。
