# SoftGPU 项目复盘

> 项目周期：PHASE1 - PHASE5 | 代码规模：~5,430 行 | Git 提交：13+ | P0/Critical Bug：0

---

## 白小东 - 协调

### 做得好

- **分阶段交付节奏把控**：将庞大项目拆解为 PHASE1-5，每个阶段目标清晰，团队始终有明确的方向，不容易迷失
- **团队沟通渠道畅通**：及时同步进度、 blockers 和需求变更，避免了信息孤岛
- **文档沉淀意识**：项目过程中持续更新设计文档和 README，降低了知识传递成本
- **风险预警及时**：遇到技术难点或延期风险时尽早暴露，没有硬扛到最后一刻

### 需改进

- **跨角色任务分配可以更细化**：偶尔出现"这件事谁来干"不够明确的情况，下次应在任务下发时就确认 A 负责、B 协助、C 审核
- **对审查意见的跟进不够紧密**：有些 code review 的 comment 提出后没有及时催办，导致问题遗留到下一阶段才发现
- **会议纪要缺失**：部分同步会没有形成书面记录，事后回顾时缺少依据

### 学到了

- 协调者的核心不是自己干得多，而是让团队干得顺——减少摩擦比亲自上场更重要
- 阶段性里程碑 + 明确的验收标准 = 团队不迷茫的前提
- 文档是最好的"时间胶囊"，哪怕是简短的 meeting notes，事后价值巨大

## 陈二虎 - 架构设计

### 做得好

- **架构设计有全局视角**：从渲染管线到内存布局，考虑了长期可扩展性，没有为了快而牺牲结构
- **抽象层次清晰**：模块边界定义合理，后期 PHASE3-5 扩展功能时无需大幅重构既有代码
- **文档与代码同步**：架构图和设计说明文档与代码实现基本一致，新成员上手有据可查
- **技术选代有理有据**：关键决策（如某算法选型）记录了 pros/cons，方便后续复盘

### 需改进

- **部分设计过于理想化**：早期架构对极端场景考虑不足，PHASE2 遇到边界条件时临时打补丁，引入了一些 tech debt
- **设计评审召集不够早**：有些关键设计在实现接近完成时才评审，修改成本偏高
- **接口契约文档不够严格**：模块间接口的输入输出约束描述不够精确，导致集成阶段出现一些对齐问题

### 学到了

- 架构设计要区分"原则性设计"和"实现细节"，前者稳住，后者允许迭代
- 早评审、快评审比晚评审、详细评审更有效——评审的价值在于 early feedback
- 接口即契约，在动手写代码前把接口签好，后面的集成工作会顺畅很多

## 白小西 - 代码实现

### 做得好

- **执行力强，交付稳定**：每个 PHASE 按计划完成功能，没有因个人原因拖期
- **代码风格一致**：整体可读性良好，变量命名和函数长度控制得当
- **主动验证**：实现完成后会用边界 case 自测，发现了不少潜在的 P0 隐患并提前修复
- **学习能力强**：新技术方案上手快，PHASE4 引入的新特性实现质量稳定

### 需改进

- **早期重实现轻重构**：部分模块初期实现较直接，后续扩展时发现结构不够灵活，调整成本较大
- **单元测试覆盖不足**：核心逻辑有测试，但边缘模块测试覆盖较薄，给后期回归带来一定风险
- **遇到 blocker 时沟通不够及时**：偶尔自己尝试太久才升级问题，下次应在 1-2 小时后主动求助

### 学到了

- "先跑通再优化"是双刃剑——跑通容易，优化难。初期多花 10-20% 时间在结构设计上，长期收益显著
- 测试是代码的"安全气囊"，不要等到出事才后悔没系上
- 遇到问题及时说，比硬撑到最后一刻成本低得多——团队协作的核心是透明

## 王刚 - 代码审查

### 做得好

- **审查覆盖面广**：不只关注 bug，也关注代码可读性、边界处理和性能隐患，真正起到了质量守门员作用
- **反馈具体且可操作**：指出的问题都有清晰的修改建议，实现者不需要反复猜测
- **审查及时**：提交后通常在当天或次日给出反馈，没有让代码等太久而阻塞后续开发
- **发现 P0 隐患**：在 PHASE3 和 PHASE5 各发现了一个潜在的数据越界问题，修复后避免了线上事故

### 需改进

- **审查粒度有时过细**：一些命名风格、注释措辞等非关键问题也提了意见，消耗了双方时间，可以考虑用 linter 统一处理
- **审查标准前后一致性可以更好**：早期审查较严格，后期某些 PR 相对宽松，导致团队对"什么是 blocker 什么是 nit"理解不一致
- **缺少对整体架构演进的审查视角**：单次 PR 审查做得不错，但缺少对多个相关 PR 合起来看的整体 review

### 学到了

- Code review 的本质是知识传递和质量保障，比例控制在 70/30（建设性反馈/挑刺）效果最佳
- 明确的 style guide + linter 能解决 80% 的 nit 问题，把人工审查精力留给真正重要的逻辑和设计
- 记录每次审查中发现的高频问题类型，团队可以针对性做 baseline training，长期能减少重复犯错

## 团队整体复盘

### 项目亮点

1. **零 P0/Critical Bug**：5 个阶段、5,430 行代码，最终 0 P0 + 0 Critical bug，质量目标达成
2. **阶段性交付稳健**：每个 PHASE 都有清晰目标，交付节奏一致，没有大的返工
3. **Git 提交规范**：13+ commits，提交粒度合理，为后续维护和追溯提供了良好基础

### 需要持续关注

1. **接口契约意识**：模块间接口文档化程度需要提高，下个项目应从一开始就把接口规范写清楚
2. **测试文化建设**：单元测试覆盖率需要提升，建议设定最低覆盖率门槛（如 70%）
3. **设计评审流程**：关键设计提前评审，不要等到实现完成再开评审会
4. **Review 标准对齐**：团队对哪些是 blocker、哪些是 nit 的理解需要统一，可以用一份 checklist 来校准

### 下一步建议

- 项目结束后安排一次 1 小时的"架构演进讨论"，梳理当前 tech debt，制定偿还计划
- 建立 team's definition of done，明确每个 PR 必须满足的条件（测试、文档、review 通过）
- 下个项目启动前，共创一份团队工作协议：如何提 PR、review 时效、blocker 升级机制等
